## 1. Что показали данные

**Бизнес:** OptCeiling — B2B продажа материалов для натяжных потолков.
**Клиенты:** монтажники, сверщики, подрядчики, ИП, ООО.
**Каналы:** Telegram (основной, 2623 чата), MAX (787), Email (316), VK (55).
**Менеджеров:** 10 активных (ОП — отдел продаж).
**Склады:** Юг, Север, Восток, Запад (Москва).

### Типичные обращения клиентов (по частоте):

## 2. Архитектура: "Сценарии" (Scenario-based Agent)

```
Клиент пишет в Telegram/MAX/VK
         │
         ▼
   ┌─────────────┐
   │  CLASSIFIER  │  ← определяет сценарий (intent)
   │  (дешёвая    │    используя Haiku/GPT-4o-mini
   │   модель)    │
   └──────┬──────┘
          │
          ▼
   ┌─────────────┐     ┌──────────────┐
   │   ROUTER    │────▶│  SCENE: ...  │
   │             │     └──────────────┘
   └─────────────┘
          │
    ┌─────┼──────┬──────────┬──────────┐
    ▼     ▼      ▼          ▼          ▼
 Сценарий  Сценарий  Сценарий  Сценарий  Сценарий
 ЗАКАЗ  ЦЕНА   НАЛИЧИЕ   НОВЫЙ     ЭСКАЛАЦИЯ
                          КЛИЕНТ    → менеджер
    │     │      │          │
    ▼     ▼      ▼          ▼
   ┌─────────────────────────┐
   │       TOOLS (MCP)       │
   │  ┌────────┐ ┌────────┐ │
   │  │  1С    │ │ Прайс  │ │
   │  │остатки │ │  лист  │ │
   │  └────────┘ └────────┘ │
   │  ┌────────┐ ┌────────┐ │
   │  │ CRM    │ │Доставка│ │
   │  │карточка│ │расчёт  │ │
   │  └────────┘ └────────┘ │
   └─────────────────────────┘
          │
          ▼
   ┌─────────────┐
   │  RESPONDER  │  ← формирует ответ (Sonnet/GPT-4o)
   │  + проверка │    НЕ выдумывает, только по данным
   └──────┬──────┘
          │
          ▼
   Ответ клиенту  ──или──  Карточка менеджеру
```

## 3. Сценарии (редактируемые конфиги)

Каждый сценарий — JSON конфиг, который можно менять без кода.

### Сценарий 1: НОВЫЙ КЛИЕНТ (квалификация)
```yaml
scene: new_client
trigger: "первое сообщение / нет в CRM / запрос прайса"
collect:
  - name: "Как вас зовут?"
  - company: "Название компании / ИП?"
  - role: "Вы монтажник, дилер или конечный заказчик?"
  - region: "В каком регионе работаете?"
  - volume: "Примерный объём закупок в месяц?"
output: "Карточка нового клиента → менеджеру"
auto_reply: false  # не отвечаем сами, передаём менеджеру
```

### Сценарий 2: ЗАКАЗ ТОВАРА
```yaml
scene: order
trigger: "список позиций с количествами"
collect:
  - items: "парсим позиции из сообщения"
  - warehouse: "С какого склада забираете?"
  - delivery: "Самовывоз или доставка?"
  - payment: "Способ оплаты?"
tools:
  - check_stock: "Проверить остатки в 1С"
  - check_price: "Получить актуальную цену"
output: "Подтверждение заказа с ценами и наличием"
escalate_if: "нет в наличии / спецраскрой / нестандарт"
```

### Сценарий 3: ПРОВЕРКА НАЛИЧИЯ
```yaml
scene: stock_check
trigger: "есть ли / в наличии / остатки"
tools:
  - check_stock: "Проверить 1С по артикулу/названию"
output: "Да, есть N единиц на складе X / Нет, ближайшая поставка..."
escalate_if: "не удалось идентифицировать товар"
```

### Сценарий 4: ЦЕНА
```yaml
scene: price
trigger: "сколько стоит / цена / прайс"
tools:
  - get_price: "Цена из 1С с учётом категории клиента (А/Б/С/S)"
output: "Цена с учётом вашей скидки: X руб/м.п."
escalate_if: "запрос индивидуальной скидки"
```

### Сценарий 5: СТАТУС ЗАКАЗА
```yaml
scene: order_status
trigger: "готов ли заказ / когда / статус"
tools:
  - get_order_status: "Статус из 1С/CRM"
output: "Ваш заказ №X: статус, ориентировочная готовность"
```

### Сценарий 6: ДОСТАВКА
```yaml
scene: delivery
trigger: "доставка / привезти / курьер"
collect:
  - address: "Адрес доставки"
  - time: "Когда удобно?"
tools:
  - calc_delivery: "Расчёт стоимости доставки"
output: "Стоимость доставки: X руб. Ближайшая дата: ..."
```

### Сценарий 7: РЕКЛАМАЦИЯ
```yaml
scene: complaint
trigger: "брак / косяк / не то прислали / возврат"
collect:
  - order_id: "Номер заказа?"
  - problem: "Что именно не так?"
  - photo: "Пришлите фото"
output: "Карточка рекламации → менеджеру + руководителю"
auto_reply: false  # ВСЕГДА передаём человеку
```

### Сценарий 8: ЭСКАЛАЦИЯ (fallback)
```yaml
scene: escalation
trigger: "не подошёл ни один сценарий / сложный вопрос"
output: "Подготовленная карточка с контекстом → менеджеру"
```

## 4. Стоимость и выбор моделей

### Двухуровневая архитектура (дёшево + качественно):

| Этап | Модель | Цена | Зачем |
|------|--------|------|-------|
| **Классификация** (intent) | Claude Haiku / GPT-4o-mini | ~$0.0003/запрос | Определить сценарий. Быстро, дёшево |
| **Генерация ответа** | Claude Sonnet / GPT-4o | ~$0.003/запрос | Сформировать ответ. Только когда нужен текст |
| **Парсинг заказа** | Claude Haiku | ~$0.0005/запрос | Извлечь позиции/количества из текста |

### Расчёт стоимости:
- Среднее сообщений в день: ~342к / 365 ≈ **940 входящих/день**
- Из них от клиентов (in): ~470/день
- Классификация: 470 × $0.0003 = **$0.14/день**
- Ответы (допустим 50% автоматизируем): 235 × $0.003 = **$0.70/день**
- **Итого: ~$25-30/месяц** на API

### Альтернатива — локальная модель:
Для классификации можно использовать fine-tuned Qwen/Llama на своём сервере → $0 за inference. Но это Phase 3.

## 5. Принцип "Не выдумывай"

Критически важно: агент НИКОГДА не придумывает данные.

```
ПРАВИЛА АГЕНТА:
1. Цены — ТОЛЬКО из 1С/прайса через tool
2. Наличие — ТОЛЬКО из 1С через tool
3. Сроки — ТОЛЬКО из CRM/производства через tool
4. Если tool не вернул данные → "Уточню у менеджера и вернусь"
5. Если клиент спрашивает то, чего нет в tools → эскалация
6. Агент НЕ обещает скидки, НЕ меняет условия, НЕ согласует возвраты
```

## 7. Интеграции (Tools/MCP)

| Tool | Источник | Что даёт |
|------|----------|----------|
| `check_stock` | 1С REST API | Остатки по артикулу и складу |
| `get_price` | 1С REST API | Цена с учётом категории клиента |
| `get_order_status` | 1С / CRM | Статус заказа |
| `calc_delivery` | Яндекс.Доставка API или таблица | Стоимость и сроки |
| `search_product` | Каталог (Elasticsearch / SQLite FTS) | Поиск по названию/артикулу |
| `get_client_info` | CRM ChatApp | История клиента, категория, долги |
| `create_card` | ChatApp API | Создать карточку/заметку для менеджера |

## 9. Технический стек

| Компонент | Технология | Почему |
|-----------|-----------|--------|
| Runtime | Python 3.12 + asyncio | Уже используем |
| LLM | Claude Haiku (classify) + Sonnet (respond) | Баланс цена/качество |
| Webhook | FastAPI | Принимаем входящие от ChatApp |
| Отправка | ChatApp API (POST messages) | Уже есть клиент |
| БД | SQLite → PostgreSQL | Начинаем просто, масштабируем |
| Сценарии | JSON конфиги | Редактируемые без кода |
| Tools | MCP серверы / простые HTTP клиенты | 1С, каталог, доставка |
| Деплой | Docker Compose | Уже настроен |

## 10. Риски и митигация

| Риск | Митигация |
|------|-----------|
| Агент ответит неправильную цену | Tools-only: цена ТОЛЬКО из 1С |
| Клиент подумает что бот | Стиль ответа = копия лучших менеджеров |
| 1С недоступна | Fallback: "Уточню и вернусь через 5 минут" |
| Сложный кейс | Быстрая эскалация + карточка для менеджера |
| Дорого | Haiku на classify ($25/мес), Sonnet только на ответы |
