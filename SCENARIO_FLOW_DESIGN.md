# ScenarioFlow — проект управляемых сценариев

**Статус:** черновик, требует обсуждения. Код не трогаем.

## Цель

Заменить текущий линейный сценарий (`response_template` + `fields[]`) на **граф узлов с условными переходами**. Агент ходит по графу детерминированно. LLM на каждом шаге видит только инструкцию текущего узла, а не весь сценарий целиком.

Это даёт:
- **Контроль** — процесс идёт строго по описанному графу, агент не "импровизирует"
- **Настраиваемость** — визуальный редактор в UI, редактируешь мышкой
- **Дешевизну** — промпты в 5–10 раз короче на каждом шаге
- **Отладку** — в UI видно, на каком узле застрял разговор

## Модель БД

### Таблица `scenario_flows`

| поле | тип | описание |
|---|---|---|
| `id` | int | PK |
| `slug` | str | уникальный, для ссылок |
| `name` | str | человеческое название |
| `trigger` | json | `{description, examples[]}` — используется в triage |
| `active` | bool | |
| `sort_order` | int | |
| `start_node_id` | str | id стартового узла |
| `nodes` | json | массив Node |
| `edges` | json | массив Edge |
| `auto_reply` | bool | если false — любой `respond` превращается в карточку менеджеру |

### Node

```json
{
  "id": "parse_sku",
  "type": "ask | tool | branch | respond | handoff",
  "label": "Распознать SKU",
  "config": { ... зависит от type ... }
}
```

### Edge

```json
{
  "from": "parse_sku",
  "to": "get_tier",
  "when": "$parse_sku.found == true",
  "label": "найдено"
}
```

`when` пустой = безусловный переход. Если у узла несколько исходящих рёбер с `when`, выполняется **первое истинное** (порядок в массиве имеет значение).

## Пять типов узлов (больше не надо)

### 1. `ask` — собрать данные у клиента

```json
{
  "type": "ask",
  "config": {
    "instruction": "Попроси клиента уточнить SKU или название профиля",
    "collect": ["sku_text"],
    "optional": false
  }
}
```

**Runtime-поведение:**
- LLM формулирует вопрос по `instruction`
- После ответа клиента LLM извлекает значения `collect`-полей
- Если все поля собраны → продвигаемся по исходящему ребру
- Если нет → остаёмся на узле, в следующем тике переформулируем вопрос

### 2. `tool` — дёрнуть инструмент (без LLM)

```json
{
  "type": "tool",
  "config": {
    "tool": "sku_recognizer",
    "args": { "text": "$sku_text" },
    "save_as": "parse_sku"
  }
}
```

**Runtime-поведение:**
- Вызов инструмента, результат кладётся в `flow_data[save_as]`
- Безусловный переход на следующий узел (если несколько рёбер — оценивается `when`)

### 3. `branch` — ветвилка по данным (без LLM)

```json
{
  "type": "branch",
  "config": {}
}
```

Пустая нода. Вся логика — в `when` на исходящих рёбрах.

### 4. `respond` — сформировать ответ клиенту

```json
{
  "type": "respond",
  "config": {
    "instruction": "Назови цену и наличие. Если профиль — предложи сопутствующие (клипсы, углы).",
    "use_data": ["sku", "price", "stock", "tier"],
    "end": true
  }
}
```

**Runtime:**
- LLM формирует ответ, видя только `instruction` + перечисленные поля
- `end: true` → после ответа флоу завершается, `current_node_id = null`
- `end: false` → продвигаемся дальше (например, `respond` + ещё один `ask`)

### 5. `handoff` — эскалация менеджеру

```json
{
  "type": "handoff",
  "config": {
    "reason": "Цена в 1С не найдена для этого SKU"
  }
}
```

Формирует карточку менеджеру, флоу завершается.

## Язык условий в `when`

Простой DSL, без Python-eval:

| синтаксис | значение |
|---|---|
| `$variable` | значение из `flow_data` или результат tool |
| `$tool.field` | поле из результата tool |
| `==`, `!=`, `>`, `<`, `>=`, `<=` | сравнение |
| `in`, `not_in` | `$tier in ['gold','platinum']` |
| `and`, `or`, `not` | логика |
| `null`, `true`, `false`, `'строка'`, `42` | литералы |

Примеры:
- `$parse_sku.found == true`
- `$tier in ['gold','platinum']`
- `$delivery_type == 'pickup' and $warehouse != null`

Парсим рекурсивным спуском, безопасно, без `eval`.

## Runtime

В `Conversation` добавляются:
- `current_flow_slug`
- `current_node_id`
- `flow_data` (json: собранные поля + результаты tool)

**Алгоритм обработки входящего сообщения:**

```
1. Triage (Haiku) — тот же флоу что current_flow_slug или новый?
   ├── тот же → current_node_id не трогаем
   └── новый → загружаем флоу, current_node_id = start_node_id, flow_data = {}

2. Если текущий узел — ask:
     а) пробуем LLM-извлечением собрать недостающие поля из нового сообщения
     б) если все собраны → продвигаемся (выбираем исходящее ребро по when)
     в) если нет → LLM формулирует переспрос, выход

3. Выполняем узлы в цикле (до 10 итераций, защита от бесконечных петель):
     tool    → дёргаем, сохраняем, идём дальше
     branch  → выбираем ребро, идём дальше
     ask     → выходим из цикла, LLM задаёт вопрос
     respond → выходим из цикла, LLM генерирует ответ
     handoff → формируем карточку, завершаем

4. Сохраняем current_node_id, flow_data, отправляем ответ
```

**Максимум узлов за один тик — 10** (предохранитель от петель).

## Пример: сценарий "Запрос цены"

Твой кейс разложенный в модель:

**Nodes:**
```
start_node_id: parse_sku

parse_sku       tool    { tool: sku_recognizer, args: {text: $message}, save_as: parse }
clarify_sku     ask     { instruction: "Не нашёл SKU, попроси уточнить", collect: [sku_text] }
get_tier        tool    { tool: crm_lookup, args: {chat_id: $chat_id}, save_as: customer }
ask_delivery    ask     { instruction: "Спроси самовывоз или доставка", collect: [delivery_type] }
ask_address     ask     { instruction: "Спроси адрес доставки", collect: [address] }
ask_warehouse   ask     { instruction: "Спроси с какого склада", collect: [warehouse] }
price_lookup    tool    { tool: onec_price, args: {sku: $parse.sku, tier: $customer.tier, ...}, save_as: price }
respond_price   respond { instruction: "Назови цену, наличие, предложи сопутствующие к профилю", end: true }
no_price        handoff { reason: "Нет цены в 1С" }
```

**Edges:**
```
parse_sku      → clarify_sku       when $parse.found == false
parse_sku      → get_tier          when $parse.found == true
clarify_sku    → parse_sku                            (безусловно, петля)
get_tier       → ask_delivery
ask_delivery   → ask_address       when $delivery_type == 'delivery'
ask_delivery   → ask_warehouse     when $delivery_type == 'pickup'
ask_address    → price_lookup
ask_warehouse  → price_lookup
price_lookup   → respond_price     when $price.success == true
price_lookup   → no_price          when $price.success == false
```

Всё. Больше никакой логики "в коде пайплайна" — всё в графе.

## UI — редактор флоу

Библиотека: **[xyflow/react-flow](https://reactflow.dev)** (MIT, бесплатная, production-ready).

**Экран:**

```
┌─────────────────────────────────────────────────────────────┐
│ Сценарии > Запрос цены                    [Тест] [Сохранить]│
├──────────┬──────────────────────────────────┬───────────────┤
│ Палитра  │                                  │ Свойства узла │
│          │    ┌───────────────┐             ├───────────────┤
│ ⬡ ask    │    │  parse_sku    │             │ ID: parse_sku │
│ ⬡ tool   │    │  (tool)       │             │ Type: tool    │
│ ⬡ branch │    └───┬───────┬───┘             │               │
│ ⬡ respond│        │       │                 │ Tool:         │
│ ⬡ handoff│  found │       │ not_found       │ [sku_recog ▼] │
│          │        ▼       ▼                 │               │
│ ──────   │  ┌────────┐  ┌─────────┐         │ Args:         │
│ Start:   │  │get_tier│  │clarify  │         │ text = $msg   │
│ parse_sku│  └────┬───┘  └────┬────┘         │               │
│          │       │           │              │ Save as:      │
│ [+ узел] │       ▼           └── петля ──┐  │ parse         │
│          │   ...                         │  │               │
│          │                               │  │ [Удалить узел]│
└──────────┴───────────────────────────────┴──┴───────────────┘
```

- **Слева** — палитра типов узлов. Drag'n'drop на холст.
- **Центр** — граф. Узлы двигаются мышью, стрелки тянутся от точки к точке, клик по стрелке → редактирование `when`.
- **Справа** — свойства выделенного узла или ребра (форма зависит от типа).
- **Сверху** — кнопка "Тест": открывает тестовый чат рядом, видно как разговор идёт по узлам подсвечивая активный.

Старый "плоский" редактор сценариев оставляем на время миграции.

## План миграции

Поэтапно, без слома существующего:

1. **Этап 1 — модель и runtime**
   - Новая таблица `scenario_flows` параллельно старой `scenes`
   - Новый модуль `agent/core/flow_runtime.py` — исполняет флоу
   - `pipeline.py` умеет оба: если `scenario_flows` непустая — используем её, иначе падаем обратно на старые `scenes`
   - Тесты руками на JSON-флоу без UI

2. **Этап 2 — базовый редактор**
   - Вкладка "Flow" в UI рядом со старым редактором
   - CRUD для `scenario_flows`
   - Read-only визуализация готовых JSON-флоу

3. **Этап 3 — интерактивный редактор**
   - Drag'n'drop узлов, рисование рёбер
   - Формы свойств для каждого типа
   - Тестовый чат с подсветкой текущего узла

4. **Этап 4 — миграция сценариев**
   - Переносим твои 10–15 сценариев вручную в новый формат
   - Выключаем старые `scenes` по одному

5. **Этап 5 — чистка**
   - Удаляем старую таблицу `scenes` и старый редактор

## Открытые вопросы (решаем до кода)

1. **Таймауты `ask`-узлов.** Что если клиент 3 дня не отвечает? Нужен ли автоматический handoff по timeout? Задаётся на узле или глобально?

2. **Повторный заход.** Клиент в середине флоу "Запрос цены", и вдруг пишет "а где склад на Лётной?". Triage определит это как отдельный запрос. Что делаем — прерываем текущий флоу, или помечаем как "подвопрос" и возвращаемся?

3. **Глобальные условия эскалации.** "Клиент ругается" или "требует менеджера" — это должно работать на ЛЮБОМ узле флоу. Где их задавать — на уровне флоу целиком как список, или это специальный узел `guard`, проверяемый перед каждым шагом?

4. **Где хранить инструменты.** Сейчас инструменты (`sku_recognizer`, `onec_price`) конфигурятся как HTTP-тулы. Оставляем так, или тоже переделываем?

5. **Версионирование флоу.** Отредактировал флоу — что с разговорами, которые сейчас в середине старой версии? Мигрируют автоматически / завершаются на старой / переносятся на аналогичный узел по id?
