# OptCeiling AI Agent — Архитектура MVP

---

## 1. Обзор системы

Система состоит из 4 компонентов:

```
┌─────────────────────────────────────────────────────────────────┐
│                        ADMIN PANEL                               │
│                                                                  │
│  Конструктор сценариев │ Настройка тона │ Реестр tools         │
└────────────────────────────┬────────────────────────────────────┘
                             │ JSON-конфиги
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                        AGENT CORE                                │
│                                                                  │
│  Triage → Tool Executor → Responder                              │
│       ▲                                           │              │
│       │           State Store (Redis)             │              │
│       └───────────────────────────────────────────┘              │
└──────────┬─────────────────────────────────┬────────────────────┘
           │                                 │
           ▼                                 ▼
┌─────────────────────┐          ┌─────────────────────────────┐
│    TEST CHAT UI      │          │       TOOL REGISTRY          │
│                      │          │                              │
│  Эмуляция диалога    │          │  1С остатки    1С цены       │
│  Прогон из истории   │          │  CRM клиент    Каталог       │
│  Сравнение с         │          │  Доставка      Оплата        │
│  реальным ответом    │          │  База знаний   ...           │
└─────────────────────┘          └─────────────────────────────┘
```

## 2. AGENT CORE — ядро системы

### 2.1 Поток обработки сообщения

```
Сообщение клиента
    │
    ▼
┌──────────────────────────────────────────────────┐
│  1. CLASSIFIER (Claude Haiku — $0.0003/запрос)   │
│                                                   │
│  Вход: сообщение + последние 5 сообщений из чата  │
│                                                   │
│  Выход:                                           │
│  {                                                │
│    "scene": "order",                              │
│    "confidence": 0.92,                            │
│    "extracted": {                                 │
│      "items": [{"name":"ПК14","qty":"3.2м"}],    │
│      "warehouse": "восток"                        │
│    }                                              │
│  }                                                │
│                                                   │
│  Если confidence < 0.7 → эскалация менеджеру     │
└──────────────────────┬───────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────┐
│  2. TRIAGE (определяет действие + сценарий)        │
│                                                   │
│  Загружает конфиг сценария из БД                  │
│  Загружает состояние диалога из Redis              │
│  Проверяет: какие данные есть, каких не хватает    │
│                                                   │
│  Решение:                                         │
│  ├─ Данных не хватает → сформировать вопрос       │
│  ├─ Данные есть → вызвать tools                   │
│  └─ Условие эскалации → передать менеджеру        │
└──────────────────────┬───────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────┐
│  3. TOOL EXECUTOR                                 │
│                                                   │
│  Берёт список tools из конфига сценария            │
│  Вызывает HTTP-запросы к внешним API              │
│  Параллельно где возможно                         │
│                                                   │
│  Если tool упал → fallback:                       │
│    "Уточню информацию и вернусь через минуту"     │
│    + уведомление менеджеру                        │
│                                                   │
│  Результат:                                       │
│  {                                                │
│    "stock": {"ПК14": 45, "warehouse": "Восток"}, │
│    "price": {"ПК14": 870}                         │
│  }                                                │
└──────────────────────┬───────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────┐
│  4. RESPONDER (Claude Sonnet — $0.003/запрос)     │
│                                                   │
│  Вход:                                            │
│  - данные от tools                                │
│  - настройки тона (из конфига)                    │
│  - примеры ответов менеджеров (few-shot)          │
│  - шаблон ответа из сценария                      │
│                                                   │
│  Формирует ответ в стиле лучших менеджеров        │
│  НЕ выдумывает данные — только из tools            │
│                                                   │
│  Выход: текст ответа клиенту                      │
└──────────────────────────────────────────────────┘
```

### 2.2 Хранение состояния (Redis)

```
Ключ: chat:{messenger}:{chat_id}

Значение:
{
  "current_scene": "order",
  "scene_data": {
    "items": [{"name": "ПК14", "qty": "3.2м"}],
    "warehouse": "восток",
    "payment": null          ← ещё не собрано
  },
  "history": [
    {"role": "client", "text": "ПК14 3.2м есть на востоке?"},
    {"role": "agent", "text": "Да, ПК14 есть — 45 шт..."}
  ],
  "client_id": "private-1370044750h...",
  "started_at": "2026-04-07T18:30:00Z"
}

TTL: 24 часа
```

Если клиент вернулся в рамках 24ч — продолжаем с того же места.
Если позже — начинаем заново.

### 2.3 Эскалация менеджеру

Когда агент не может обработать сам, он формирует карточку:

```
┌──────────────────────────────────────────┐
│  КАРТОЧКА ДЛЯ МЕНЕДЖЕРА                  │
│                                          │
│  Клиент: Барэмбо Игорь Сергеевич        │
│  Телефон: +7 915 019 42 42              │
│  Мессенджер: Telegram                    │
│  Категория: (С)                          │
│                                          │
│  Сценарий: Оформление заказа             │
│  Причина эскалации: Клиент просит скидку │
│                                          │
│  Собранные данные:                       │
│  ✅ Товары: ПК14 3.2м, тандем 5м        │
│  ✅ Склад: Восток                        │
│  ✅ Цена: 2 784₽ (по прайсу)            │
│  ❌ Оплата: не уточнена                  │
│                                          │
│  Диалог:                                 │
│  > ПК14 3.2м, тандем 5м, с востока      │
│  < Есть, итого 2 784₽. Как оплатите?    │
│  > А скидку дадите? Я постоянный        │
│  → ЭСКАЛАЦИЯ                             │
└──────────────────────────────────────────┘
```

Карточка отправляется как заметка в ChatApp чат.
Менеджер видит полный контекст и продолжает диалог.

## 3. КОНСТРУКТОР СЦЕН

### 3.1 Структура сценария (JSON)

```json
{
  "id": "order",
  "name": "Оформление заказа",
  "active": true,

  "trigger": {
    "description": "Клиент присылает список товаров с количествами или просит оформить заказ",
    "keywords": ["заказ", "нужен", "выпишите", "запустите", "закажите"],
    "examples": [
      "ПК14 3.2м, гарпун тандем 5м",
      "Запустите пожалуйста бауф черный",
      "Нужен профиль EuroKraab 80 метров"
    ]
  },

  "fields": [
    {
      "name": "items",
      "type": "product_list",
      "required": true,
      "prompt": "Какие позиции вам нужны?"
    },
    {
      "name": "warehouse",
      "type": "enum",
      "options": ["юг", "север", "восток", "запад"],
      "required": true,
      "prompt": "С какого склада забираете?"
    },
    {
      "name": "payment",
      "type": "enum",
      "options": ["карта", "наличные", "счёт", "QR"],
      "required": true,
      "prompt": "Как оплачиваете?"
    },
    {
      "name": "delivery",
      "type": "text",
      "required": false,
      "prompt": "Нужна доставка? Если да — укажите адрес"
    }
  ],

  "tools": [
    {
      "tool": "search_product",
      "when": "items заполнены, но не распознаны как артикулы",
      "args": {"query": "$items[].name"}
    },
    {
      "tool": "check_stock",
      "when": "items и warehouse заполнены",
      "args": {"items": "$items", "warehouse": "$warehouse"}
    },
    {
      "tool": "get_price",
      "when": "items заполнены",
      "args": {"items": "$items", "client_id": "$client.id"}
    }
  ],

  "response_template": "Подтверди каждую позицию с ценой и наличием. Назови итоговую сумму. Спроси способ оплаты если не указан.",

  "escalate_when": [
    "клиент просит скидку",
    "товара нет ни на одном складе",
    "спецраскрой или нестандартный размер",
    "клиент недоволен или конфликтует"
  ]
}
```

## 5. РЕЕСТР ИНСТРУМЕНТОВ (TOOLS)

### 5.1 Структура инструмента (JSON)

```json
{
  "id": "check_stock",
  "name": "Проверка остатков",
  "description": "Проверяет наличие товара на складе через 1С",
  "active": true,

  "request": {
    "method": "GET",
    "url": "https://1c.optceiling.ru/api/v1/stock",
    "headers": {
      "Authorization": "Basic ${env.1C_AUTH}"
    },
    "params": {
      "article": "${args.article}",
      "warehouse": "${args.warehouse}"
    }
  },

  "response_mapping": {
    "quantity": "$.data.quantity",
    "price": "$.data.price",
    "warehouse_name": "$.data.warehouse.name",
    "available": "$.data.quantity > 0"
  },

  "fallback": "Не удалось проверить остатки. Уточню у менеджера.",
  "timeout_ms": 5000,
  "retry": 1
}
```

> Глобальные знания хранятся в сценарии `faq`, при ответе мержатся с
> знаниями текущего сценария и подставляются напрямую в системный промпт.

## 6. TEST CHAT UI

### 6.1 Назначение

Веб-страница для тестирования агента до подключения к реальным мессенджерам.
Три режима работы:

**Режим 1 — Ручной чат.** Ты пишешь как клиент, агент отвечает.
Видишь весь pipeline: какой сценарий, какие tools вызвал, какой ответ.

**Режим 2 — Прогон из истории.** Берём реальный диалог из нашей базы
(342к сообщений). Подаём сообщения клиента агенту. Сравниваем ответ
агента с реальным ответом менеджера. Считаем процент совпадений.

**Режим 3 — Теневой режим.** Агент подключён к реальному потоку
сообщений, но НЕ отправляет ответы. Менеджер отвечает сам.
Мы видим рядом: что ответил менеджер и что ответил бы агент.

## 7. ИСПОЛЬЗОВАНИЕ СОБРАННЫХ ДАННЫХ (342 289 сообщений)

Данные НЕ используются для fine-tune модели.
Используются для 5 конкретных задач:

### 7.1 Эталонные ответы для тона

```
Источник: ответы топ-5 менеджеров (54 000+ сообщений)
Результат: 30-50 отобранных примеров "вопрос → идеальный ответ"
Применение: few-shot примеры в промпте Responder
Формат: JSON массив в конфиге тона
```

### 7.2 Каталог товаров

```
Источник: все упоминания товаров в сообщениях клиентов
Результат: словарь товаров с вариантами написания
  "ПК14" → ["пк14", "пк-14", "ПК 14", "pk14"]
  "BORZZ Light 30" → ["борзз лайт", "борз лайт 30", "borzz light"]
Применение: tool search_product (нечёткий поиск)
Формат: SQLite FTS таблица
```

### 7.3 Автоматическое тестирование

```
Источник: 1000 случайных диалогов из базы
Процесс:
  1. Берём сообщение клиента
  2. Прогоняем через агента
  3. Сравниваем ответ агента с реальным ответом менеджера
  4. Считаем метрики: правильный сценарий? правильные данные?
Применение: CI/CD — при каждом изменении сценария прогоняем тесты
```

### 7.4 Обучение классификатора

```
Источник: 500 входящих сообщений, размеченных по сценариям
Процесс:
  1. Берём случайные входящие сообщения
  2. Вручную размечаем: "это заказ", "это наличие", "это цена"
  3. Прогоняем через классификатор
  4. Считаем accuracy (цель: >95%)
  5. Если ошибается — правим промпт классификатора
Применение: тестовый набор для качества классификации
```

### 7.5 База знаний (FAQ)

```
Источник: повторяющиеся ответы менеджеров
Результат: структурированные ответы на частые вопросы
  - Условия бесплатной доставки
  - Адреса складов и время работы
  - Как оплатить по QR
  - Условия возврата
  - Минимальная сумма заказа
Применение: встроена в сценарии (поле knowledge), подаётся в системный промпт респондера
Формат: JSON-массив [{question, answer}] в каждом сценарии
```
